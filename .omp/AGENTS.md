# AI-native development workflow

Этот каталог описывает современный подход к разработке через нейросети: не "попросить AI написать код", а собрать управляемый цикл из ролей, проверок и коротких итераций.

## Принцип

AI ускоряет исследование, прототипирование и ревью, но ответственность остается у разработчика. Поэтому каждый AI-цикл должен оставлять проверяемый след:

- цель изменения;
- список измененных файлов;
- команды проверки;
- известные риски;
- решение: принять, доработать или откатить.

## Роли

- `agents/maker.md` — исследует задачу, предлагает минимальный план и вносит изменения.
- `agents/checker.md` — проверяет решение как reviewer: ищет регрессии, слепые зоны, проблемы безопасности и тестовые пробелы.
- `skills/loop-demo/SKILL.md` — демонстрирует полный цикл Maker -> Checker -> Fix -> Verify.

## Рабочий цикл

1. Сформулировать задачу одним абзацем: что должно измениться для пользователя или разработчика.
2. Запустить Maker: прочитать контекст, выбрать минимальный план, внести изменения.
3. Запустить Checker: проверить diff, тесты, edge cases и соответствие архитектуре проекта.
4. Исправить замечания, которые меняют поведение или надежность.
5. Выполнить локальную верификацию.
6. Зафиксировать итог: что изменено, чем проверено, что осталось риском.

## Definition of Done

Изменение считается готовым, если:

- оно решает исходную задачу без лишнего расширения scope;
- код следует текущим паттернам проекта;
- есть релевантная проверка: тест, build, lint или ручная проверка сценария;
- все AI-предложения были проверены человеком или вторым агентом;
- в финальном отчете названы ограничения и непроверенные зоны.

## Команды проекта

Используйте эти команды как базовую сетку проверок:

```bash
npm run test
npm run build
```

Для контентных изменений достаточно:

```bash
npm run content
npm run index
```

## Ограничения

AI не должен без ручной проверки менять:

- авторизацию и обработку сессий;
- платежи, персональные данные, секреты и инфраструктурные ключи;
- миграции базы данных;
- production deploy scripts.

В этих областях агент может готовить черновик, список рисков и тест-план, но решение принимает разработчик.
