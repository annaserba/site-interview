## Короткий ответ

Интервьюер ждёт именно техническую сложность, а не организационную. Выберите задачу с жёсткими инженерными ограничениями: высокая нагрузка, нестандартный алгоритм, сложная интеграция, работа с большими данными в реальном времени. Структура: технический контекст и узкое место → почему наивное решение не проходило → какие альтернативы рассмотрели → реализованное решение с деталями → измеримый результат.

## Контекст

Уточнение «технически сложную» — прямой сигнал не уходить в менеджерские кейсы. В Яндексе и Лиге Ставок проверяют глубину инженерной экспертизы.

## Как строить ответ

### Очертить техническую проблему

Объясните ограничения: производительность, память, latency, CAP-теорема, конкурентность, объём данных. Почему готовые решения не подходили.

### Разобрать альтернативы

Какие подходы отвергли и почему. Покажите инженерный trade-off анализ.

### Детализировать реализацию

Назовите конкретные структуры данных, алгоритмы, архитектурные паттерны, которые применили.

### Подтвердить цифрами

Latency до/после, объём обработанных данных, нагрузка, количество ошибок.

## Пример ответа

Самый технически сложный проект — миграция монолитного Angular-приложения на React с Micro Frontends. Масштаб: 500K LOC, 20 разработчиков, 3 команды. Сложность: 1) Параллельная работа; 2) Shared state — данные корзины доступны из обоих приложений; 3) Performance — bundle не должен вырасти. Решение: 1) Module Federation (Webpack 5); 2) Custom event bus; 3) Gradual migration — по одному page per sprint. Личный вклад: спроектировал архитектуру shared layer, написал Migration script, провёл 15 tech interviews. Результат: миграция за 8 месяцев (план 12), zero downtime, bundle -30%.

## Частые ошибки

- Рассказывать про конфликт команды или сжатые сроки вместо технической проблемы.
- Не объяснять, почему простые решения не работали.
- Сказать «использовали готовую библиотеку» без объяснения выбора.
- Не называть конкретные метрики до и после.
