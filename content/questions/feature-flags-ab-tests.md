## Короткий ответ

Feature Flag (фича-флаг) — механизм включения/выключения функциональности без деплоя нового кода. Код деплоится заранее в выключенном состоянии, а включение происходит через конфигурацию. A/B-тест — это частный случай фича-флагов, где две версии показываются разным группам пользователей для сравнения метрик.

## Контекст

Интервьюер проверяет понимание современных практик деплоя. Ожидается знание того, как фича-флаги декоррелируют деплой и релиз, позволяют делать canary-релизы, A/B-тесты и мгновенный rollback.

## Как строить ответ

### Что такое Feature Flag
Фича-флаг — это условный оператор в коде, управляемый через конфигурацию (LaunchDarkly, Unleash, собственное решение). Код с новой функциональностью деплоится на прод в выключенном состоянии. Включение — изменение конфига, не требующее деплоя.

### Связь с A/B-тестами
A/B-тест — это фича-флаг, который показывает разные версии разным сегментам пользователей. Например, 50% видят старую версию, 50% — новую. Затем сравниваются метрики (конверсия, retention). Ключевое: рандомизация должна быть стабильной (user_id, не session_id).

### Преимущества
- **Декорреляция деплоя и релиза** — код на проде, но не активен
- **Мгновенный rollback** — выключение флага за секунды
- **Canary releases** — постепенное включение (1% → 10% → 50% → 100%)
- **Trunk-based development** — все коммитят в main, фичи скрыты флагами

### Техническая реализация
```javascript
if (featureFlags.isEnabled('new-checkout', user.id)) {
  return <NewCheckout />
}
return <OldCheckout />
```

### Очистка флагов
Флаги должны быть временными. После полного раскатывания фичи флаг и старый код удаляются. Иначе накапливается technical debt.

### Частые ошибки
- Забывать удалять флаги после раскатки — код становится неподдерживаемым
- Использовать session_id вместо user_id для A/B — пользователь видит разные версии
- Не мониторить метрики при включении флага
- Не иметь kill switch для аварийного выключения
