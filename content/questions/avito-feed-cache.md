## Короткий ответ

Разделите entity-кеш (объявления по id) и выдачу (cursor+список id). Ключ — viewer, фильтры и версия алгоритма. Отдавайте кеш сразу с conditional ревалидацией, ограничивайте память LRU.

## Что ожидают в ответе

TTL — страховка, а не основной механизм корректности. Необходимо разделение entity cache и представлений, контроль памяти через LRU-бюджет, защита от race conditions при смене фильтров и обработка мутаций через optimistic update с reconcile.

## Контекст

Пользователь листает cursor-based персонализированную ленту и возвращается назад без скачка позиции. Объявления обновляются и удаляются, порядок выдачи меняется, а вкладка может жить часами при ограниченной памяти мобильного устройства.

## Как строить ответ

### Разделить сущности и представления

Храните канонические объявления в `entities[id]`, а результат выдачи — как последовательность id и cursor. Версия или updatedAt предотвращает перезапись нового объекта устаревшим ответом другой страницы.

### Спроектировать ключ и консистентность

Ключ включает viewer, фильтры, сортировку, географию, эксперимент и версию ranking. SWR улучшает latency, но сервер остаётся источником истины. Для удалений используйте tombstone, для обновлений — версионирование, для conditional GET — ETag или revision.

### Контролировать память и навигацию

Задайте лимит по числу сущностей или байтам, вытесняйте далёкие страницы по LRU и сохраняйте anchor id плюс offset отдельно. Не рассчитывайте только на TTL: длинная сессия способна накопить сотни страниц до его истечения.

### Обработать мутации и гонки

Optimistic update должен иметь mutation id, rollback и reconcile с серверной версией. Отменяйте устаревшие запросы при смене фильтров, дедуплицируйте in-flight fetch и не позволяйте медленному старому ответу перезаписать новую выдачу.

### Измерять эффективность

Смотрите hit ratio вместе с freshness, объёмом памяти, числом сетевых запросов, временем возврата к ленте и долей некорректных карточек. Высокий hit ratio бесполезен, если пользователь видит удалённые объявления.

## Пример ответа

Для кеша ленты из миллионов объявлений я использую многоуровневую архитектуру. На клиенте — IndexedDB для хранения закэшированных объявлений с TTL, а localStorage — для метаданных пагинации. При загрузке ленты сначала проверяем кеш: если данные актуальны, отдаём из IndexedDB, иначе делаем запрос к API и обновляем кеш. Для оптимизации памяти используем виртуализацию списка через react-window — рендерим только видимые элементы. Ключевой подход — нормализация данных: храним объявления в Map по ID, а ленту — как массив ID. Это позволяет переиспользовать данные между страницами. Для синхронизации между вкладками используем Broadcast Channel API.

## Частые ошибки

- Кешировать персональную выдачу без viewer и версии ranking в ключе.
- Использовать TTL как гарантию консистентности.
- Хранить полные копии объявлений в каждой странице.
- Не защищаться от гонки ответов после смены фильтров.
- Вытеснять данные, не сохранив anchor для восстановления скролла.
