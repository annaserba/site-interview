## Короткий ответ

Проверьте не только видимые циклы, но и стоимость операций внутри них. `[...result, item]`, `unshift()` и обычно `shift()` требуют копирования или перемещения O(k) элементов. Если k растёт на каждой из n итераций, суммарная работа равна 1 + 2 + … + n = O(n²). Исправления: `push()` с единственным `reverse()` в конце, очередь через head-индекс и хранение parent-ссылок вместо копий путей.

## Контекст

Код внешне содержит один цикл, но на больших данных резко замедляется и потребляет много памяти. Нужно найти линейные операции стандартной библиотеки внутри цикла, вывести суммарную сложность и предложить структуру данных под реальные операции.

## Как строить ответ

### Считать стоимость каждой операции

Не считайте все методы массива O(1). `push` и `pop` амортизированно константны, а вставка или удаление в начале массива требуют сдвига элементов. Spread создаёт новый контейнер и копирует ссылки; стоимость зависит от текущей длины.

### Просуммировать растущую работу

Если на итерации i копируется i элементов, один внешний цикл не делает алгоритм линейным: сумма арифметической прогрессии даёт O(n²). Пиковая память может выглядеть O(n), но общий объём аллокаций и давление на GC также становятся квадратичными.

### Выбрать представление под паттерн доступа

Для очереди используйте массив и head-индекс либо deque. Для построения результата в обратном порядке — `push`, затем один `reverse`. Для маршрутов в графе храните предка вершины, а не отдельный массив полного пути для каждой вершины.

### Подтвердить гипотезу измерениями

Big O отвечает за рост, но не заменяет профилирование. Сравните n, 2n и 4n на репрезентативных данных, снимите CPU profile, число аллокаций и паузы GC. Учитывайте конкретный движок, но не основывайте корректность SLA на недокументированной оптимизации `shift`.

## Пример ответа

Скрытая квадратичная сложность часто возникает с includes, indexOf, splice в цикле:

```javascript
// O(n²) — splice в цикле
function removeDuplicates(arr) {
  const result = [];
  for (const item of arr) {
    if (!result.includes(item)) { // O(n) поиск
      result.push(item);
    }
  }
  return result;
}

// O(n) — используем Set
function removeDuplicates(arr) {
  return [...new Set(arr)];
}
```

Другие примеры: concat в цикле (создаёт новый массив каждый раз), object spread в цикле. На практике: использую Set для дедупликации, Map для группировки, избегаюincludes в циклах. Для проверки множеств — O(1) lookup в Set/Map vs O(n) в Array.

## Частые ошибки

- Оценивать только количество явно написанных циклов.
- Считать spread синтаксическим сахаром без стоимости копирования.
- Менять `shift` на `pop`, не проверив требуемый порядок FIFO.
- Оптимизировать микробенчмарк, не измеряя аллокации и GC в реальной нагрузке.
- Утверждать, что конкретный JavaScript-движок всегда выполняет `shift()` за O(1).
