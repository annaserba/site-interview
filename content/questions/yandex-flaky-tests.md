## Короткий ответ

Flaky тесты — это тесты, которые проходят и падают без изменений в коде. Причины: race conditions, внешние зависимости (сеть, БД), тайминги, гонки потоков, неочищенные данные. Методы борьбы: изоляция тестов, retry, wait strategies, очистка состояния, аналитика CI для трекинга flaky rate.

## Контекст

Интервьюер хочет убедиться, что вы понимаете проблему flaky tests на практике, а не только понаслышке. Ожидается знание конкретных причин, методов детекции и политик компании по борьбе с ними.

## Как строить ответ

### Определение и масштаб проблемы

Flaky тест — тест, который даёт разные результаты (pass/fail) при одинаковом коде. Это разрушает доверие к CI: разработчики начинают игнорировать падения. Исследования показывают, что flaky тесты замедляют.delivery на 10-15% из-за повторных запусков и ручных проверок.

### Основные причины

Race conditions: тест зависит от порядка выполнения или асинхронных операций. Внешние зависимости: сеть, БД, API, файловая система. Тайминги: `setTimeout`, `waitFor` с фиксированными задержками. Состояние: тесты делят данные, не очищают после себя. Окружение: разные версии ОС, браузеров, окружения CI.

### Методы борьбы

Иоляция: каждый тест создаёт и очищает свои данные, не зависит от других. Retry: повторный запуск упавшего теста (осторожно — маскирует реальные баги). Wait strategies: вместо `sleep(1000)` — polling с таймаутом, ожидание конкретного условия. Порядок: сортировка тестов, детерминированный порядок выполнения. Очистка: beforeEach/afterEach для сброса состояния.

### Детекция и политика

Аналитика CI: трекинг flaky rate по тестам, алерты при превышении порога. Quarantine: flaky тесты отключаются до исправления. Policy: flaky тесты блокируют merge, требуют исправления. Культура: flaky тест — баг, который нужно чинить, а не игнорировать.

## Код из интервью

```javascript
// ❌ Плохо: фиксированный sleep — источник flakiness
test('element appears', async () => {
  await page.click('#load-button');
  await page.waitForTimeout(2000); // Может не успеть или ждать лишнего
  expect(await page.$('.result')).toBeTruthy();
});

// ✅ Хорошо: wait strategy — ожидание конкретного условия
test('element appears', async () => {
  await page.click('#load-button');
  await expect(page.locator('.result')).toBeVisible({ timeout: 5000 });
});

// ✅ Хорошо: изоляция данных через factory
beforeEach(async () => {
  this.user = await UserFactory.create(); // Уникальный пользователь
});

afterEach(async () => {
  await User.delete(this.user.id); // Очистка
});
```

## Пример ответа

Flaky тесты — серьёзная проблема, потому что подрывают доверие к CI/CD.

**Причины на практике:** Самые частые — race conditions в асинхронном коде (тест «жалеет» данные до завершения операции) и внешние зависимости (тест падает, потому что API вернул 500). Также бывают гонки между тестами (один тест меняет данные, которые использует другой) и фиксированные `sleep()` вместо ожидания условия.

**Методы борьбы:** Изоляция — каждый тест работает со своими данными и очищает за собой. Wait strategies — вместо `setTimeout` используем `waitFor` с конкретным условием. Retry — как защитный механизм, но не как решение. Аналитика — трекинг flaky rate в CI, алерты при превышении 1-2%.

**Политика компании:** В идеале — flaky тест блокирует merge, требует немедленного исправления. На практике — quarantine для критических flaky, постепенное исправление. Важно не маскировать проблему retry'ем, а устранять корень.

**Измерение:** Считаем flaky rate = количество flaky запусков / общее количество запусков. Цель: < 0.5% flaky rate. Мониторим в CI dashboard, алертируем команду при росте.

## Частые ошибки

- Использовать retry как основной метод борьбы — это маскирует реальные баги.
- Игнорировать flaky тесты — «они иногда падают, это нормально» разрушает доверие к CI.
- Не трекать flaky rate — без метрик невозможно улучшить ситуацию.
- Использовать `sleep()` вместо ожидания условия — главный источник flakiness в UI-тестах.
