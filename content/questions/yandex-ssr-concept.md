## Короткий ответ

SSR (Server-Side Rendering) — рендеринг HTML на сервере перед отправкой клиенту. Нужен для SEO (поисковики видят готовый HTML), ускорения первой отрисовки (FCP) и работы без JavaScript. Альтернативы: CSR (рендер на клиенте) и SSG (генерация при сборке). Современные реализации: Next.js, Nuxt, streaming SSR, React Server Components.

## Контекст

Интервьюер хочет понять, почему SSR нужен, а не просто «так делают». Ожидается сравнение SSR/CSR/SSG, знание гидратации и trade-offs (нагрузка на сервер, сложность debugging).

## Как строить ответ

### SSR vs CSR vs SSG

CSR: сервер отдаёт пустой `<div id="app">`, JS рендерит всё на клиенте. Быстрая навигация, но медленный FCP и плохой SEO. SSR: сервер рендерит HTML, клиент «оживляет» его (гидратация). Быстрый FCP, SEO, но нагрузка на сервер. SSG: HTML генерируется при сборке (build time). Максимально быстро, но не подходит для динамических данных.

### Когда использовать SSR

SEO-зависимые страницы (лендинги, каталоги, новостные сайты). Страницы с динамическими данными, которые нужно показать сразу (лендинги с персонализацией). Медленные клиенты (мобильные сети, старые устройства). Когда нужен Open Graph / sharing (соцсети парсят HTML).

### Как работает гидратация

После SSR клиент получает HTML + JSON-сниппет (`__NEXT_DATA__` или аналог). React/Nuxt «приклеивает» обработчики событий к существующему HTML без пересоздания DOM. hydration mismatch (серверный HTML ≠ клиентский рендер) — частая проблема. Streaming SSR (React 18, Next.js App Router) — построчная отправка HTML для ускорения FCP.

### Trade-offs и современные альтернативы

Нагрузка на сервер: SSR требует серверных вычислений для каждого запроса (caching!). Сложность: два контекста выполнения (сервер/клиент), разные API (fs, window). React Server Components (Next.js App Router) — рендеринг компонентов на сервере без отправки JS-бандла клиенту. Islands Architecture (Astro, Fresh) — серверный рендеринг с гидратацией только интерактивных частей.

## Код из интервью

```javascript
// Пример SSR на Next.js (Pages Router)
// pages/products/[id].js

export async function getServerSideProps(context) {
  const { id } = context.params;
  const res = await fetch(`https://api.example.com/products/${id}`);
  const product = await res.json();

  return {
    props: { product }, // Передаётся как JSON в __NEXT_DATA__
  };
}

export default function ProductPage({ product }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price}</p>
    </div>
  );
}

// Next.js автоматически:
// 1. Вызывает getServerSideProps на сервере
// 2. Рендерит HTML с данными
// 3. Отдаёт HTML + __NEXT_DATA__ клиенту
// 4. Клиент «оживляет» компонент (гидратация)
```

## Пример ответа

SSR — это подход, где HTML страницы формируется на сервере и отправляется клиенту в готовом виде.

**Зачем:** Во-первых, SEO — поисковые роботы сразу видят контент. Во-вторых, быстрая первая отрисовка — пользователь видит контент до загрузки JS-бандла. В-третьих, соцсети корректно отображают превью (og:tags). В-четвёртых, работает на устройствах с отключённым JS.

**Как это работает:** SSR выполняет React-компоненты на сервере, сериализует результат в HTML + JSON. Клиент получает HTML (виден сразу) и JSON с данными. При загрузке React «гидрирует» DOM — навешивает обработчики без пересоздания элементов.

**Альтернативы:** CSR проще в разработке, но хуже для SEO и FCP. SSG идеален для статических страниц, но не для динамических данных. Современные подходы: streaming SSR (построчная отправка), React Server Components (рендер без JS), Islands (гидратация только интерактивных частей).

**Trade-offs:** Нужен сервер с вычислительной мощностью. Сложнее debugging (два контекста). Кэширование SSR-страниц критически важно для производительности.

## Частые ошибки

- Считать, что SSR решает все проблемы производительности — он ускоряет FCP, но не общее время загрузки.
- Забывать про гидратацию — mismatch между серверным и клиентским рендером приводит к багам.
- Использовать SSR для всех страниц без необходимости — статические страницы лучше SSG.
- Не кэшировать SSR-ответы — каждый запрос к серверу создаёт нагрузку.
