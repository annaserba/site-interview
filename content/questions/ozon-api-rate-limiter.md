## Короткий ответ

Подход зависит от требований к всплескам и распределённости. Для одного процесса подойдут token bucket и очередь с ограниченным числом воркеров. В Go удобно использовать канал как семафор, context для отмены и ticker для пополнения токенов. Для нескольких инстансов состояние лимита выносят в Redis или централизованный gateway.

## Контекст

Нужно защитить внешний API от превышения квоты, но не терять входящие запросы и корректно обрабатывать отмену и остановку приложения.

## Как строить ответ

### Уточнить модель лимита

Определите лимит, допустимый burst, максимальное ожидание и область действия: процесс, пользователь или весь кластер.

### Разделить очередь и допуск

Ограниченная очередь создаёт backpressure, а token bucket решает, когда следующий запрос можно отправить.

### Корректно завершаться

Используйте context для таймаутов и отмены, закрывайте воркеры без паники и возвращайте понятную ошибку при переполнении очереди.

## Частые ошибки

- Запускать отдельную goroutine на каждый запрос без ограничения.
- Использовать ticker без остановки и получить утечку ресурсов.
- Забыть, что локальный лимитер не координирует несколько инстансов.
