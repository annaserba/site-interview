## Короткий ответ

PR → 1-2 ревьюера → комментарии → исправления → мерж. Важно: не блокировать, давать обратную связь, учитьjunior-ов. Инструменты: GitHub PR, GitLab MR.

## Контекст

Проверяют понимание процесса code review и его важности для качества кода.

## Как строить ответ

### Процесс

Опишите flow: от создания PR до мержа.

### Ревьюеры

Кто и сколько ревьюит. Как распределяете нагрузку.

### Обратная связь

Как даёте комментарии: конструктивно, с примерами.

## Пример ответа

Процесс: PR → 1-2 ревьюера → комментарии → исправления → мерж. Ревьюеры: senior-ы дляjunior-ов, peers для peers. Время: не более 24 часов на ревью. Комментарии: конструктивные, с примерами кода. Результат: снижение багов на 40%, ростjunior-ов.

## Частые ошибки

- Не ревьюить вообще
- Ревьюить только формально
- Не давать обратную связьjunior-ам
- Блокировать PR без веских причин
